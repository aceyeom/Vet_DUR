## 1. System Overview

The Vet DUR system is built as **5 independent, composable analysis engines**, each evaluating a distinct clinical risk domain. When a DUR scan is triggered, all engines run concurrently and return typed alert arrays. The final safety score is a weighted composite across all five engines.

### Processing Flow

```
Pharmacist clicks "Execute DUR Scan"
        │
        ▼
┌─────────────────────────────────────────────┐
│   Patient Profile + Prescription Data       │
└──────┬──────┬──────┬──────┬─────────────────┘
       │      │      │      │      │
       ▼      ▼      ▼      ▼      ▼
    [DDI] [Dose] [Allergy] [Disease] [Duplicate]
       │      │      │      │      │
       └──────┴──────┴──────┴──────┘
                      │
                      ▼
            Score Aggregator
            (weighted composite)
                      │
                      ▼
            SafetyReview Component
```

1. Prescription data + patient profile passed to all 5 engines simultaneously
2. Each engine queries its own database tables independently
3. Each engine returns a structured alert array: `{ type, severity, title, shortFix, action }`
4. Score aggregator weights alerts by severity and engine priority
5. Combined result returned to the frontend `SafetyReview` component

### Engine Priority & Score Weights

| Engine | Name | Weight | Layer | Data Source |
|--------|------|--------|-------|-------------|
| 1 | Drug-Drug Interaction (DDI) | **35%** | Backend (Python) | `ddi_pairs` + `drugs` |
| 2 | Dosage Verification | **25%** | Backend (Python) | `dosing_rules` |
| 3 | Allergy Checking | **20%** | Frontend (JS) | `allergy_crossref` |
| 4 | Disease Contraindication | **15%** | Backend + RAG | `contraindications` + RAG |
| 5 | Drug Duplication | **5%** | Frontend (JS) | `drug_dictionary` |

### Shared Alert Schema

Every engine returns alerts in this format:

```javascript
{
  type: "Interaction" | "Dose" | "Allergy" | "Disease" | "Duplicate",
  severity: "Critical" | "Warning" | "Info",
  title: "string",
  shortFix: "string",
  mechanism: "string",          // optional — DDI and Disease engines
  evidenceLevel: "A" | "B" | "C" | "D",  // optional
  sources: ["string"],          // optional — RAG-retrieved citations
  action: {
    type: "replace" | "update" | "remove",
    target: "DrugName",
    value: "NewDrugOrDose"
  }
}
```

## 2. Engine 1 — Drug-Drug Interaction (DDI)

**Purpose:** Detect clinically significant interactions between any two or more drugs in the active prescription, including pharmacokinetic (CYP-mediated) and pharmacodynamic (additive toxicity) interactions.

**Layer:** Python backend  
**Weight:** 35%

### Architecture

The DDI engine uses a **two-pass approach**:

- **Pass 1 — Rule-based pair lookup:** For every drug pair in the prescription, query `ddi_pairs` for a known interaction record.
- **Pass 2 — CYP inference:** For pairs not found in `ddi_pairs`, cross-reference each drug's CYP metabolism profile to infer potential pharmacokinetic interactions.
- **Multi-drug scoring:** Scores from all pairs are aggregated using a max-plus-penalty model — the worst single pair sets the floor, each additional moderate+ interaction adds a diminishing penalty.

### Database Tables

#### `ddi_pairs`

Core interaction knowledge base. Each row is a known pairwise drug-drug interaction. Records are either **empirical** (sourced from clinical literature) or **inferred** (generated by the CYP/transporter RAG pipeline).

### Scoring Logic

#### Pass 1 — Empirical Pair Scores

| Condition | Points | Alert Level |
|-----------|--------|-------------|
| Contraindicated pair | +100 | Critical |
| Major pair | +75 | Critical |
| Moderate pair | +40 | Warning |
| Minor pair | +15 | Info |
| Each additional moderate+ pair (beyond first) | +50% of base score (diminishing) | — |

---

#### Pass 2 — Inferred Pair Scores
*No explicit record exists — interaction predicted from drug attribute overlap*

| Mechanism | Base Score | Confidence Multiplier | Final Score | Alert Level |
|-----------|-----------|----------------------|-------------|-------------|
| CYP substrate + strong inhibitor (same isoform) | 70 | × 0.60 | **42** | Warning |
| CYP substrate + moderate inhibitor | 45 | × 0.60 | **27** | Warning |
| CYP substrate + inducer | 30 | × 0.60 | **18** | Info |
| P-gp substrate + P-gp inhibitor | 55 | × 0.60 | **33** | Warning |
| High protein binding competition (both >90%, one NTI) | 45 | × 0.60 | **27** | Warning |
| Electrolyte depletion + NTI cardiac drug | 70 | × 0.60 | **42** | Warning |

---

#### Pass 3 — Pharmacodynamic Overlap Scores
*Additive toxicity — independent of metabolic mechanism*

| Condition | Base Score | Patient Risk Escalation | Alert Level |
|-----------|-----------|------------------------|-------------|
| 2 drugs with same `moderate` PD flag | +40 | × 1.0 (healthy patient) | Warning |
| 2 drugs with same `high` PD flag | +65 | × 1.0 (healthy patient) | Critical |
| Any PD overlap + patient has related condition | base score | × 1.5–2.0 | Escalated |
| NTI drug involved in any overlap | base score | × 1.3 | Escalated |

**Patient risk multipliers by condition:**

| Patient Condition | Relevant PD Flag | Multiplier |
|-------------------|-----------------|------------|
| Renal failure / CKD | `nephrotoxicity` | × 1.8 |
| Cardiac disease / arrhythmia | `qt_prolongation` | × 2.0 |
| Liver disease / elevated ALT | `hepatotoxicity` | × 1.8 |
| Epilepsy / neurological disease | `seizure_threshold_effect` | × 1.5 |
| Hypotension / shock | `hypotension` | × 1.6 |
| Active bleeding / coagulopathy | `bleeding_risk` | × 1.8 |

## 3. Engine 2 — Dosage Verification

**Purpose:** Validate each drug's dose against species-specific therapeutic ranges, adjusted for patient weight, renal function, hepatic function, age, and breed-specific metabolism differences.

**Layer:** Python backend  
**Weight:** 25%

### Architecture

1. Fetch `dosing_rules` record for each drug + species combination
2. Calculate actual dose (`prescription dosage × patient weight`)
3. Apply adjustment factors: renal impairment factor (creatinine-based), hepatic factor (ALT-based), age factor (pediatric/geriatric multipliers)
4. Compare adjusted range against prescribed dose
5. Flag underdose (below min), overdose (above max), or inappropriate interval

### Database Tables

#### `dosing_rules`

| Column | Type | Description |
|--------|------|-------------|
| `id` | UUID / PK | Primary key |
| `drug_id` | FK → drugs | Drug reference |
| `species` | VARCHAR | `dog` \| `cat` \| `horse` \| `cattle` etc. |
| `indication` | VARCHAR | What condition this dosing is for |
| `min_mg_per_kg` | DECIMAL | Minimum therapeutic dose |
| `max_mg_per_kg` | DECIMAL | Maximum safe dose |
| `min_interval_hr` | INT | Minimum dosing interval in hours |
| `max_interval_hr` | INT | Maximum dosing interval |
| `route` | VARCHAR | `oral` \| `IV` \| `SC` \| `IM` \| `topical` |
| `narrow_therapeutic_index` | BOOLEAN | If true, flag any deviation immediately |
| `pediatric_max_mg_per_kg` | DECIMAL | Override for young/juvenile animals |
| `geriatric_adjustment_pct` | DECIMAL | % reduction for geriatric patients (e.g. `0.75` = 25% reduction) |
| `source` | TEXT | Reference for this dosing data |

#### `renal_dose_adjustments`

| Column | Type | Description |
|--------|------|-------------|
| `drug_id` | FK → drugs | Drug reference |
| `species` | VARCHAR | Species this applies to |
| `creatinine_mild_max` | DECIMAL | Upper creatinine threshold for mild impairment (mg/dL) |
| `mild_factor` | DECIMAL | Dose multiplier for mild impairment (e.g. `0.75`) |
| `creatinine_moderate_max` | DECIMAL | Upper creatinine threshold for moderate impairment |
| `moderate_factor` | DECIMAL | Dose multiplier for moderate impairment (e.g. `0.5`) |
| `severe_factor` | DECIMAL | Dose multiplier for severe — `0` means contraindicated |
| `renal_elimination_pct` | DECIMAL | % of drug eliminated renally |

### Scoring Logic

| Condition | Points | Alert Level |
|-----------|--------|-------------|
| Dose > max (NTI drug) | +60 | Critical |
| Dose > max (standard drug) | +40 | Warning |
| Dose < min | +30 | Warning |
| Renal adjustment required but not applied | +50 | Critical |
| Interval too frequent | +30 | Warning |

## 4. Engine 3 — Allergy Checking

**Purpose:** Cross-reference the patient's known allergens against all prescribed drugs and their drug classes, including cross-reactive drug families. A dog allergic to Penicillin must also be flagged for Amoxicillin and Clavulox.

**Layer:** Frontend (JavaScript) — no backend call needed  
**Weight:** 20%

### Architecture

1. Load `patient.allergies` array (e.g. `['Penicillin', 'Sulfonamides']`)
2. For each prescribed drug, look up its `allergy_class` and cross-reactive classes
3. Check if drug itself OR any cross-reactive class matches a patient allergen
4. Flag all matches as Critical — allergy alerts are **never** downgraded to Warning

### Database Tables

#### `allergy_crossref`

| Column | Type | Description |
|--------|------|-------------|
| `drug_id` | FK → drugs | Drug reference |
| `allergy_class` | VARCHAR | e.g. `beta-lactam`, `sulfonamide`, `fluoroquinolone` |
| `cross_reactive_with` | TEXT[] | Array of allergy classes this drug cross-reacts with |
| `cross_reactivity_strength` | ENUM | `high` \| `moderate` \| `low` |
| `notes` | TEXT | e.g. `'Cephalosporins ~2% cross-react with Penicillin'` |

#### `patient_allergies` *(future — when patient records are persisted)*

| Column | Type | Description |
|--------|------|-------------|
| `patient_id` | FK → patients | Patient reference |
| `allergen_name` | VARCHAR | Drug or class name (normalized to `allergy_class`) |
| `reaction_type` | VARCHAR | `anaphylaxis` \| `urticaria` \| `GI` \| `unknown` |
| `severity` | ENUM | `life-threatening` \| `severe` \| `moderate` \| `mild` |
| `confirmed` | BOOLEAN | Clinician-confirmed vs. owner-reported |

### Scoring Logic

| Condition | Points | Alert Level |
|-----------|--------|-------------|
| Direct drug allergen match | +100 | Critical (hard block) |
| Cross-reactive class match (high strength) | +90 | Critical |
| Cross-reactive class match (moderate) | +60 | Critical + note |
| Cross-reactive class match (low) | +30 | Warning |

## 5. Engine 4 — Disease Contraindication

**Purpose:** Flag drugs that are contraindicated or require caution given the patient's active disease conditions — including genetic conditions (MDR1/ABCB1 mutation), organ dysfunction (renal failure, liver disease), and breed-specific risks.

**Layer:** Backend + RAG  
**Weight:** 15%

### Architecture

Two-layer approach:

- **Layer 1 — Rule-based lookup:** Query `contraindications` table for explicit drug + condition pairs. Fast, deterministic, covers known contraindications.
- **Layer 2 — RAG augmentation:** For complex multi-condition scenarios or gaps in explicit rules, query the vector store and retrieve relevant literature chunks. Layer 2 only runs if Layer 1 produces 0 alerts OR patient has 2+ conditions.

### Database Tables

#### `contraindications`

| Column | Type | Description |
|--------|------|-------------|
| `id` | UUID / PK | Primary key |
| `drug_id` | FK → drugs | Drug being contraindicated |
| `condition_name` | VARCHAR | Normalized condition string (e.g. `'MDR1 Deficiency'`) |
| `condition_tags` | TEXT[] | Searchable tags: `['mdr1', 'abcb1', 'gene_mutation']` |
| `contraindication_type` | ENUM | `absolute` \| `relative` \| `caution` |
| `severity` | ENUM | `critical` \| `major` \| `moderate` \| `minor` |
| `mechanism` | TEXT | Why this drug is dangerous in this condition |
| `species` | TEXT[] | Species this applies to — null means all |
| `breed_specific` | TEXT[] | e.g. `['Shetland Sheepdog', 'Collie', 'Australian Shepherd']` |
| `alternative_drug_ids` | UUID[] | Suggested safer alternatives from the formulary |
| `rag_chunk_ids` | TEXT[] | Vector store chunks for cited explanation generation |

#### `condition_synonyms`
Normalizes free-text patient history to canonical condition names.

| Column | Type | Description |
|--------|------|-------------|
| `canonical_name` | VARCHAR | Normalized name used in `contraindications` table |

### RAG Pipeline (Layer 2)

```
Query construction:
  "Patient: [species] [breed] [conditions]. Drug: [name]. Contraindication?"
        │
        ▼
  Vector store retrieval (top 3–5 chunks)
  [vet case reports, pharmacology papers, breed-sensitivity literature]
        │
        ▼
  LLM synthesis (Gemini/GPT)
  generates clinical explanation citing retrieved chunks
        │
        ▼
  Appended to rule-based alert as "explanation" field
```

## 6. Engine 5 — Drug Duplication

**Purpose:** Detect therapeutic duplicates — two or more prescribed drugs in the same therapeutic class providing overlapping clinical effects without additive benefit, increasing toxicity risk.

**Layer:** Frontend (JavaScript)  
**Weight:** 5%

### Architecture

1. For each drug pair, compare `primary_class` values
2. If two drugs share the same class, flag as duplication
3. Cross-check `duplication_exceptions` — some same-class combos are intentional (e.g. dual antibiotics for synergistic spectrum)
4. Also check for same active ingredient under different brand names

### Database Tables

#### `therapeutic_classes` *(extension of drug_dictionary)*

| Column | Type | Description |
|--------|------|-------------|
| `drug_id` | FK → drugs | Drug reference |
| `primary_class` | VARCHAR | Top-level class: `NSAID`, `Antibiotic`, `Opioid`, `ACE Inhibitor` etc. |
| `subclass` | VARCHAR | Finer classification: `beta-lactam`, `fluoroquinolone`, `mu-opioid` etc. |
| `active_ingredient` | VARCHAR | INN name — catches brand/generic duplicates |
| `is_combination_product` | BOOLEAN | True if drug contains multiple active ingredients (e.g. Clavulox) |
| `component_drug_ids` | UUID[] | Component drug IDs for combination products |

#### `duplication_exceptions`

| Column | Type | Description |
|--------|------|-------------|
| `drug_a_id` | FK → drugs | First drug |
| `drug_b_id` | FK → drugs | Second drug |
| `rationale` | TEXT | Why this combination is intentional |

### Scoring Logic

| Condition | Points | Alert Level |
|-----------|--------|-------------|
| Same `primary_class` | +50 | Warning |
| Same `active_ingredient` (brand/generic duplicate) | +80 | Critical |
| Component overlap with combination product | +50 | Warning |
| Pair found in `duplication_exceptions` | 0 | No alert |

## 8. DDI Algorithm Deep Dive

### Key Drug Attributes

Before the algorithm runs, it needs the following attributes per drug, sourced from `drugs` and `drug_cyp_profiles`:

| Attribute | What It Means | How It's Used in DDI |
|-----------|--------------|----------------------|
| **CYP substrate** | Drug is broken down by this enzyme | If another drug inhibits that enzyme, levels accumulate dangerously |
| **CYP inhibitor** | Drug blocks this enzyme (weak/moderate/strong) | Raises plasma levels of co-prescribed substrates on same isoform |
| **CYP inducer** | Drug accelerates this enzyme | Lowers levels of co-prescribed substrates — risk of underdose |
| **Protein binding %** | How much drug binds to plasma proteins | Two highly-bound drugs compete for binding sites, raising free drug levels |
| **Renal elimination %** | How much is cleared by kidneys | Compounds risk in patients with renal failure |
| **Half-life** | How long drug stays in system | Longer half-life = longer interaction window |
| **QT prolongation risk** | Can extend cardiac QT interval | Two QT-prolonging drugs together → potentially fatal arrhythmia |
| **Nephrotoxicity risk** | Can damage kidneys | Additive toxicity risk — especially dangerous with renal failure patients |
| **Hepatotoxicity risk** | Can damage liver | Additive with impaired hepatic clearance |
| **CNS depression risk** | Sedation/respiratory depression | Additive with other CNS depressants |
| **Narrow therapeutic index (NTI)** | Small margin between effective and toxic dose | Any interaction immediately becomes highest priority |

### Two-Pass Algorithm

#### Pass 1 — Explicit Pair Lookup

```python
from itertools import combinations

def pass_one_explicit_lookup(prescription, db):
    alerts = []
    drug_pairs = list(combinations(prescription, 2))

    for drug_a, drug_b in drug_pairs:
        # Always query with alphabetical ordering (how ddi_pairs stores them)
        a, b = sorted([drug_a, drug_b], key=lambda d: d['name'])

        result = db.query("""
            SELECT * FROM ddi_pairs
            WHERE drug_a_id = %s AND drug_b_id = %s
        """, [a['id'], b['id']])

        if result:
            alerts.append({
                "type": "Interaction",
                "severity": "Critical" if result.severity in ["contraindicated", "major"] else "Warning",
                "title": f"{drug_a['name']} + {drug_b['name']}",
                "mechanism": result.mechanism,
                "shortFix": result.management,
                "severity_score": result.severity_score,
                "evidence_level": result.evidence_level,
                "sources": result.source_refs,
                "action": build_action(result, drug_a, drug_b)
            })

    return alerts
```

#### Pass 2 — CYP Inference

```python
def pass_two_cyp_inference(prescription, db, known_pair_ids):
    alerts = []
    drug_pairs = list(combinations(prescription, 2))

    for drug_a, drug_b in drug_pairs:
        # Skip pairs already caught in pass 1
        if (drug_a['id'], drug_b['id']) in known_pair_ids:
            continue

        cyp_a = db.query("SELECT * FROM drug_cyp_profiles WHERE drug_id = %s", [drug_a['id']])
        cyp_b = db.query("SELECT * FROM drug_cyp_profiles WHERE drug_id = %s", [drug_b['id']])

        interaction = check_cyp_interaction(cyp_a, cyp_b)

        if interaction:
            # CYP-inferred alerts are scored at 60% of their inferred severity
            inferred_score = interaction['base_score'] * 0.60
            alerts.append({
                "type": "Interaction",
                "severity": "Warning",
                "title": f"Potential CYP Interaction: {drug_a['name']} + {drug_b['name']}",
                "mechanism": interaction['mechanism'],
                "severity_score": inferred_score,
                "evidence_level": "D",  # Theoretical — no explicit record
                "action": { "type": "update", "target": drug_a['name'], "value": None }
            })

    return alerts


def check_cyp_interaction(cyp_a, cyp_b):
    """
    Core CYP inference logic.
    Returns interaction dict if found, None otherwise.
    """
    for profile_a in cyp_a:
        for profile_b in cyp_b:
            if profile_a.cyp_isoenzyme != profile_b.cyp_isoenzyme:
                continue

            # Substrate + Inhibitor on same isoform = accumulation risk
            if profile_a.role == "substrate" and profile_b.role == "inhibitor":
                strength_score = {"strong": 70, "moderate": 45, "weak": 20}
                base = strength_score.get(profile_b.inhibition_strength, 20)
                return {
                    "mechanism": (
                        f"{profile_b.drug_id} inhibits {profile_a.cyp_isoenzyme}, "
                        f"reducing clearance of {profile_a.drug_id} "
                        f"— potential accumulation to toxic levels"
                    ),
                    "base_score": base
                }

            # Substrate + Inducer on same isoform = underdose risk
            if profile_a.role == "substrate" and profile_b.role == "inducer":
                return {
                    "mechanism": (
                        f"{profile_b.drug_id} induces {profile_a.cyp_isoenzyme}, "
                        f"accelerating clearance of {profile_a.drug_id} "
                        f"— potential subtherapeutic levels"
                    ),
                    "base_score": 30
                }

    return None
```

#### Pharmacodynamic Overlap Check

```python
ADDITIVE_RISK_FLAGS = [
    "qt_prolongation_risk",
    "nephrotoxicity",
    "hepatotoxicity",
    "bleeding_risk",
    "cns_depression_risk",
    "gi_ulcer_risk",
]

def check_pharmacodynamic_overlap(prescription, patient, db):
    alerts = []

    for flag in ADDITIVE_RISK_FLAGS:
        # Find all drugs in prescription with this risk flag set to moderate or high
        at_risk = [
            drug for drug in prescription
            if get_risk_level(drug, flag, db) in ["moderate", "high"]
        ]

        if len(at_risk) >= 2:
            # Escalate score if patient already has a related condition
            patient_multiplier = get_patient_risk_multiplier(patient, flag)
            base_score = 40 * len(at_risk) * patient_multiplier

            alerts.append({
                "type": "Interaction",
                "severity": "Critical" if base_score >= 60 else "Warning",
                "title": f"Additive {flag.replace('_', ' ').title()}: {', '.join(d['name'] for d in at_risk)}",
                "mechanism": f"Multiple drugs with {flag} risk prescribed concurrently",
                "severity_score": min(base_score, 100),
                "action": { "type": "remove", "target": at_risk[-1]['name'], "value": None }
            })

    return alerts


def get_patient_risk_multiplier(patient, flag):
    """
    Escalates score when patient already has a condition that
    makes the pharmacodynamic risk worse.
    e.g. nephrotoxic drugs in a patient with renal failure = 1.8x multiplier
    """
    multipliers = {
        "nephrotoxicity": {
            "condition_keywords": ["renal failure", "kidney disease", "CKD"],
            "multiplier": 1.8
        },
        "hepatotoxicity": {
            "condition_keywords": ["liver disease", "hepatic failure", "elevated ALT"],
            "multiplier": 1.8
        },
        "qt_prolongation_risk": {
            "condition_keywords": ["arrhythmia", "cardiac disease", "heart disease"],
            "multiplier": 2.0
        },
        "cns_depression_risk": {
            "condition_keywords": ["seizure", "neurological", "epilepsy"],
            "multiplier": 1.5
        },
    }

    config = multipliers.get(flag)
    if not config:
        return 1.0

    for condition in patient.get("conditions", []):
        if any(kw.lower() in condition.lower() for kw in config["condition_keywords"]):
            return config["multiplier"]

    return 1.0
```

### Multi-Drug Score Aggregation

```python
def aggregate_ddi_score(alerts):
    """
    Max-plus-penalty model:
    - Worst single pair sets the floor
    - Each additional moderate+ pair adds 50% of its score (diminishing returns)
    - Prevents a 10-drug prescription from hitting 1000% while still
      reflecting that more interactions = more danger
    """
    if not alerts:
        return 0

    scores = sorted([a["severity_score"] for a in alerts], reverse=True)

    total = scores[0]  # Worst pair sets the floor
    for i, score in enumerate(scores[1:], start=1):
        if score >= 40:  # Only penalize moderate+ interactions
            total += score * (0.5 ** i)  # Diminishing returns

    return min(int(total), 100)
```

## 9. Testing Strategy

### Accuracy Expectations

The rule-based system's accuracy is **bounded entirely by data coverage**, not algorithm logic. Realistic targets by phase:

| Phase | Expected Recall | What's Covered |
|-------|----------------|----------------|
| Early (sparse `ddi_pairs`) | 40–60% | Major/well-documented interactions only |
| With CYP inference layer | +15–20% | Metabolic pairs without explicit records |
| Mature database | 85–90% | Comparable to human-medicine DDI checkers |

> ⚠️ **The dangerous failure mode is false negatives** (missing real interactions), not false positives (over-alerting). The algorithm never misses what's in the database — it only misses what isn't there yet.

### Metrics to Track Over Time

| Metric | Target | What It Tells You |
|--------|--------|-------------------|
| **Recall** | ≥ 80% | % of real interactions caught — primary clinical safety metric |
| **Precision** | ≥ 70% | % of fired alerts that are real — tracks alert fatigue risk |
| **False negative rate** | < 20% | Directly maps to patient safety gap |
| **CYP inference contribution** | Track separately | Shows how much the inference layer covers your data gap |
| **Alert resolution rate** | > 60% | % of alerts pharmacists actually act on — low = alert fatigue |
