# 데이터 구조 비교: JSONL vs DB Schema

## 1. 현재 JSONL 구조 (plumbs_structured_db.jsonl)

### 최상위 필드
```json
{
  "ingredient": "string",      // 약물 성분명 (예: Granisetron)
  "source_file": "string",     // PDF 파일명 (예: 7.pdf)
  "sections": {
    // 중첩된 텍스트 필드들
  }
}
```

### Sections 필드 (nested JSON)
```
- General Information
- Uses/Indications
- Pharmacology/Actions
- Pharmacokinetics
- Contraindications/Precautions/Warnings
- Adverse Effects
- Reproductive/Nursing Safety
- Overdose/Acute Toxicity
- Drug Interactions
- Laboratory Considerations
- Dosages
- Monitoring
- Client Information
- Chemistry/Synonyms
- Storage/Stability
- Compatibility/Compounding Considerations
- Dosage Forms/Regulatory Status
```

### 특징
- ✅ **계층적 비정규화 구조** - 관련 정보를 함께 저장
- ✅ **원본 텍스트 보존** - 추출 소스를 그대로 유지
- ❌ **검색 어려움** - 특정 정보를 찾으려면 텍스트 파싱 필요
- ❌ **중복 가능성** - 여러 JSON에서 같은 정보 중복 가능
- ❌ **구조화되지 않음** - 자유형식 텍스트

## 2. 목표 DB Schema (정규화된 구조)

### 핵심 테이블 맵핑

#### A. 기본 정보 → substances + data_sources
```
JSONL: ingredient
   ↓
DB: substances table
   - inn_name (약물명)
   - korean_name
   - drug_class
   - classification_code
   
JSONL: source_file + General Information
   ↓
DB: data_sources table
   - source_type: 'plumbs'
   - source_name: 'Plumb\'s Veterinary Drug Handbook'
   - accessed_at
```

#### B. Uses/Indications → therapeutic_classes
```
JSONL sections.Uses/Indications
   ↓
DB: therapeutic_classes table
   - primary_class: "Antiemetic", "5-HT3 Antagonist"
   - subclass
```

#### C. Pharmacokinetics → pk_parameters
```
JSONL sections.Pharmacokinetics
   ↓
DB: pk_parameters table
   - substance_id (FK)
   - species: 'dog', 'cat', 'horse', etc
   - route: 'oral', 'IV', 'IM', 'SC'
   - half_life_hr
   - bioavailability_pct
   - time_to_peak_hr
   - protein_binding_pct
   - volume_of_distribution_l_per_kg
```

#### D. Adverse Effects → pd_risk_flags
```
JSONL sections.Adverse Effects
   ↓
DB: pd_risk_flags table
   - substance_id (FK)
   - nephrotoxicity: 'none' | 'low' | 'moderate' | 'high'
   - hepatotoxicity
   - qt_prolongation
   - myelosuppression
   - cns_depression
   - respiratory_depression
   - adverse_effects_dog (TEXT)
   - adverse_effects_cat (TEXT)
```

#### E. Drug Interactions → ddi_pairs
```
JSONL sections.Drug Interactions
   ↓
DB: ddi_pairs table
   - substance_a_id (FK)
   - substance_b_id (FK)
   - interaction_type
   - severity: 'contraindicated' | 'major' | 'moderate' | 'minor'
   - mechanism_detail
   - clinical_effect
```

#### F. Contraindications → contraindications
```
JSONL sections.Contraindications/Precautions/Warnings
   ↓
DB: contraindications table
   - substance_id (FK)
   - condition_name
   - contraindication_type: 'absolute' | 'relative' | 'caution'
   - mechanism
```

#### G. Dosages → dosing_rules
```
JSONL sections.Dosages
   ↓
DB: dosing_rules table
   - substance_id (FK)
   - species: 'dog', 'cat'
   - route: 'oral', 'IV', 'IM', 'SC'
   - indication
   - min_dose
   - max_dose
   - maintenance_interval_hr
   - loading_dose (if applicable)
```

#### H. Reproductive/Nursing Safety → pd_risk_flags
```
JSONL sections.Reproductive/Nursing Safety
   ↓
DB: pd_risk_flags table
   - reproductive_safety_class_papich: 'A' | 'B' | 'C' | 'D'
   - reproductive_safety_note
```

#### I. Overdose/Acute Toxicity → pd_risk_flags
```
JSONL sections.Overdose/Acute Toxicity
   ↓
DB: pd_risk_flags table
   - overdose_ld50_mg_per_kg
   - overdose_toxic_dose_mg_per_kg
   - overdose_signs
   - overdose_treatment
```

#### J. Chemistry/Synonyms → substance_synonyms
```
JSONL sections.Chemistry/Synonyms
   ↓
DB: substance_synonyms table
   - substance_id (FK)
   - synonym: "Kytril", "BRL-43694A", etc
   - synonym_type: 'brand' | 'inn' | 'generic' | 'abbreviation'
   - language: 'ko' | 'en' | 'la'
```

#### K. 원본 텍스트 보존 → literature_chunks
```
JSONL sections.* (모든 텍스트)
   ↓
DB: literature_chunks table
   - substance_id (FK)
   - pmc_id (if applicable)
   - chunk_text: 원본 텍스트
   - field_group: 'pharmacokinetics', 'adverse_effects', etc
   - embedding: vector(1536) - 의미 검색용
```

## 3. 비교표

| 측면 | JSONL | DB Schema |
|------|-------|----------|
| **구조** | 비정규화 | 정규화 (3NF) |
| **레코드당 크기** | 대 (완전한 데이터) | 소 (특정 정보만) |
| **중복 제거** | ❌ 불가능 | ✅ 가능 (정규화) |
| **검색 성능** | ❌ 느림 (텍스트 파싱) | ✅ 빠름 (인덱스) |
| **트랜잭션** | ❌ 불가능 | ✅ ACID 준수 |
| **원본 텍스트** | ✅ 보존 | ✅ literature_chunks 별도 저장 |
| **쿼리 유연성** | ❌ 제한적 | ✅ SQL 조인 가능 |
| **데이터 무결성** | ❌ FK 제약 없음 | ✅ FK 제약 있음 |
| **스키마 진화** | ❌ 어려움 | ✅ Migration 가능 |

## 4. 데이터 마이그레이션 전략

### Phase 1: 데이터 추출 및 매핑
```
JSONL 읽기
   ↓
sections 파싱
   ↓
각 필드를 DB 테이블에 맵핑
   ↓
텍스트 처리 (임베딩 생성 등)
```

### Phase 2: ETL 파이프라인
```python
1. substances 테이블에 ingredient 삽입
2. data_sources에 source_file 정보 삽입
3. sections별로 해당 테이블에 따로 저장:
   - Pharmacokinetics → pk_parameters
   - Adverse Effects → pd_risk_flags
   - Drug Interactions → ddi_pairs
   - Dosages → dosing_rules
   - etc.
4. 모든 원본 텍스트 → literature_chunks
```

### Phase 3: 데이터 검증
```
- FK 참조 무결성 확인
- 데이터 완성도 검사
- 중복 데이터 제거
- 데이터 타입 검증
```

## 5. 결론

### 현재 상태
- **용도**: 원본 PDF 데이터의 **임시 저장 및 분석**
- **형식**: Plumb's Veterinary Drug Handbook에서 자동 추출
- **역할**: **RAW 데이터 → 전처리 단계**

### 목표 상태
- **용도**: DUR 엔진의 **정규화된 데이터 저장소**
- **형식**: PostgreSQL 관계형 데이터베이스
- **역할**: **구조화된 데이터 → 분석 및 쿼리 가능**

3. ✅ 데이터 검증 및 품질 체크
4. ✅ 약물 상호작용 추론 엔진 활성화
